## T149 — Presence + Awareness (7 tasks)

### Dependency Graph

```
T146 (CRDT WS sync, W1)
  └─► T256  T149.1: Awareness WS handler (parse + broadcast)
        ├─► T257  T149.2: SDK API (setLocalAwarenessState, onAwarenessChange, getAwarenessStates)
        │     ├─► T267  T149.5: Integration test (3 agents, distinct cursors)
        │     └─► T271  T149.7: Docs page (awareness how-to)
        └─► T258  T149.3: Presence registry (in-memory, 30s TTL, 10s sweep)
              └─► T266  T149.4: REST GET /api/v1/documents/:slug/presence
                    ├─► T268  T149.6: Frontend PresenceIndicator component
                    └─► T271  T149.7: Docs page (also depends on T257)
```

### Design Decisions (T149)

- **Protocol**: y-sync `messageAwareness = 1`. Server relays raw bytes without decoding — peers decode client-side via `y-protocols/awareness`. No server-side state parsing.
- **Persistence**: None. Ephemeral in-memory `PresenceRegistry`. Survives process restart via client reconnect within 5s.
- **TTL**: 30 seconds per entry. Expiry sweep every 10 seconds via `setInterval`. No Redis dependency for single-instance deployment.
- **REST surface**: `GET /api/v1/documents/:slug/presence` returns `{agentId, section, cursorOffset?, lastSeen}[]`. Requires auth.
- **Frontend**: Polling component (10s interval). WS-driven push deferred as a future enhancement.

### Validation Gate

`cleo orchestrate ready --epic T149` returns 0 tasks because T256 correctly depends on T146 (not yet shipped). Once T146 completes, T256 becomes the first ready task.

## T150 — Turn-Taking Leases (8 tasks)

### Dependency Graph

```
T148 (event log, W1)
  │
  ├─► T277  T150.1: schema — section_leases table migration  [IVTR READY NOW]
  │     └─► T278  T150.2: Lease service (acquireLease, renewLease, releaseLease, getActiveLease)
  │           ├─► T279  T150.3: REST endpoints (POST/DELETE/GET/PATCH /lease)
  │           │     ├─► T285  T150.6: SDK LeaseManager (acquire, renew, release, autoRenew)
  │           │     ├─► T288  T150.7: Integration test (2-agent contention)
  │           │     └─► T289  T150.8: Docs page (leases how-to)
  │           ├─► T282  T150.4: Event emission (SECTION_LEASED/RELEASED/EXPIRED via T148)
  │           │     └─► T288  (also depends here)
  │           └─► T284  T150.5: TTL expiry job (15s interval, emits SECTION_LEASE_EXPIRED)
  │                 └─► T288  (also depends here)
  └─(T282, T284 both depend on T148 for insertEvent)
```

### Design Decisions (T150)

- **Lease semantics**: ADVISORY, not mandatory. CRDT (T146) still accepts writes from non-holders. The 409 `SECTION_LEASED` response is a cooperative signal — not an enforcement barrier. Byzantine resistance is T152's problem.
- **Max TTL**: 300 seconds (5 minutes) enforced in route input validation. Prevents indefinite hold.
- **Auto-renew**: `LeaseManager.startAutoRenew(thresholdSeconds)` prevents accidental expiry during long edits.
- **Event integration**: All three lease state changes (`SECTION_LEASED`, `SECTION_LEASE_RELEASED`, `SECTION_LEASE_EXPIRED`) emit into the T148 `document_events` table so subscribers automatically see lease lifecycle.
- **Expiry job**: 15-second `setInterval`. Cleared on server shutdown.
- **Schema**: New `section_leases` table with `(docId, sectionId)` composite index. Migration generated by drizzle-kit but not applied in T150.1 — apply in IVTR phase.

### Validation Gate

`cleo orchestrate ready --epic T150` returns T277 immediately. T277 has zero within-W2 dependencies and can be started as soon as an IVTR worker is spawned.

## T151 — Differential Subscriptions (9 tasks)

### Dependency Graph

```
T148 (event log, W1)           T146 (CRDT, W1, for delta format context)
  │                               │
  ├─► T293  T151.1: path-matcher  [ZERO DEPS — IVTR READY NOW]
  │
  ├─► T294  T151.2: diff helper (getEventsSince, computeSectionDelta)
  │     ├─► T295  T151.3: SSE subscribe endpoint (also needs T293, T148)
  │     │     └─► T301  T151.5: SDK subscribe() method
  │     │           ├─► T304  T151.7: Integration test (path isolation)
  │     │           └─► T306  T151.9: Docs page
  │     └─► T299  T151.4: Section GET ?since=seq delta mode
  │           └─► T302  T151.6: SDK fetchSectionDelta()
  │                 ├─► T304  (also depends here)
  │                 ├─► T305  T151.8: Bandwidth regression test
  │                 └─► T306  (also depends here)
  │
  (T295 depends on T293 + T294 + T148)
```

### Design Decisions (T151)

- **Transport**: Reuses T148 event log + existing SSE infrastructure. No new fanout layer, no Redis pub/sub added. T077 (Redis event bus) may enhance throughput in a future pass but is not required for correctness.
- **Two surfaces**: (1) `?since=seq` query param on existing `GET /sections/:name` — backward compatible additive change; (2) `GET /api/v1/subscribe?path=` SSE endpoint with path-pattern filtering.
- **Diff mode**: `Accept: application/vnd.llmtxt.diff+json` content negotiation triggers `computeSectionDelta` inclusion in SSE payload.
- **Path matcher**: Supports `*` (single segment wildcard) and `:param` (named capture). Implemented as pure regex with zero external deps.
- **Resume**: `Last-Event-ID` HTTP header sets resume position. Events with `seq <= Last-Event-ID` are not re-delivered.
- **Bandwidth gate**: T305 is a mandatory CI test asserting delta responses are at least 5x smaller than full responses for small edits to large documents.
- **No-op 200**: `GET /sections/:name?since=N` when `currentSeq=N` returns `{delta: null, currentSeq: N}` with 200 (not 304) to simplify client polling logic.

## Open Design Questions

1. **Multi-instance presence**: The in-memory `PresenceRegistry` does not survive horizontal scaling (multiple backend pods). Redis pub/sub would be needed. Deferred to a future task; single-instance Railway deployment is the current target.

2. **Lease enforcement vs. advisory**: T150 makes leases advisory. If future MA-8 (Byzantine resistance) needs hard enforcement, it must be added as a separate layer — do not retrofit T150.

3. **SSE backpressure**: `GET /api/v1/subscribe` holds connections open indefinitely. A connection limit or idle timeout should be added if load tests reveal resource exhaustion. This is a T160 (load tests) concern.

4. **SectionDelta scope in T151**: The `computeSectionDelta` implementation returns deltas derived from `document_events` event log entries. It does NOT compute text-level diffs (line adds/removes). For character-level diffs, Yrs binary updates (T146) are the right surface. T151 is event-granularity, not character-granularity.

5. **y-protocols version**: T149 assumes `y-protocols` is available in `packages/llmtxt`. If T146 did not add it, T149.1's IVTR worker must add it as a `pnpm` dependency first.
